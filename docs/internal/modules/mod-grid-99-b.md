# 문서·시연 인프라 모듈 (`apps/docs` 외)

`@topgrid` 그리드 제품의 **문서 사이트 · 컴포넌트 시연 · 시각 회귀 · 마이그레이션 가이드 ·
패키지 README**를 묶는 인프라 모듈. 그리드 런타임 코드가 아니라, 13개 `@topgrid/grid-*`
패키지를 외부에 노출·검증·안내하는 도구 체인을 정의한다.

- 위치: `apps/docs` (private, 외부 배포 대상 아님) + 모노레포 루트의 `playwright.config.ts` /
  `tests/visual/` / `.github/workflows/`
- 런타임 번들 영향: **0 KB** — 전부 devDependencies(문서 빌드/CI 전용), 그리드 패키지
  `src/` 미변경.
- 라이선스 정책: 도입 도구는 모두 MIT 또는 Apache-2.0 (상용 SaaS 의존 회피).

대상 13개 패키지: `grid`(meta), `grid-core`, `grid-features`, `grid-export`,
`grid-renderers`, `grid-license`, `grid-pro-agg`, `grid-pro-datamap`, `grid-pro-header`,
`grid-pro-master`, `grid-pro-merging`, `grid-pro-range`, `grid-pro-tracking`.

## 2. API 문서 사이트 (Docusaurus + TypeDoc)

### 2.1 핵심 설계

- **정적 사이트 생성기: Docusaurus v3** (`@docusaurus/core ^3`, preset-classic).
  React 기반 + MDX + pnpm workspace 호환이 채택 근거. Vite 모노레포에 Next.js를 추가로
  들이는 Nextra, Vue 기반이라 기여 장벽이 있는 VitePress를 모두 배제.
- **TypeDoc 연동: `docusaurus-plugin-typedoc`** (내부적으로 `typedoc` +
  `typedoc-plugin-markdown`). 13개 패키지의 `src/index.ts`를 entryPoint로 받아
  TypeScript export → Markdown으로 변환하고 Docusaurus 사이드바 "API Reference"
  섹션에 자동 등록한다. 수동 MDX 변환(13개 × 갱신 주기 유지비)·HTML iframe 삽입(검색/테마
  통합 불가)을 배제한 결과.
- **`entryPointStrategy: 'packages'`** — 모노레포의 워크스페이스 경로를 자동 해석.
- **`failOnWarnings: true`** (`typedoc.json`) — 공개 API에 JSDoc 누락 시 빌드를 실패로
  처리. 문서화 의무를 CI에서 강제하는 게이트.

### 2.2 설정 계약

```typescript
// docusaurus.config.ts (요지 — 실제 config 기준)
const config: Config = {
  onBrokenLinks: 'throw',
  onBrokenMarkdownLinks: 'warn',
  i18n: { defaultLocale: 'ko', locales: ['ko', 'en'] },
  presets: [
    ['classic', {
      docs: { sidebarPath: './sidebars.ts', routeBasePath: '/' },
      blog: false,
      // `theme` 키 없음 — 글로벌 CSS 미생성, preset 기본 테마로 빌드(설계결정 §6.1).
    } satisfies Preset.Options],
  ],
  // NOTE(실제): `plugins` 배열 자체가 없다 — 자동 API 레퍼런스
  // (docusaurus-plugin-typedoc)는 typedoc 버전 정합 이슈로 임시 비활성이며,
  // 복구 시 아래 형태로 추가한다. `out: 'api'` + sidebars.ts 의 autogenerated 'api'.
  //   plugins: [['docusaurus-plugin-typedoc', { entryPoints, entryPointStrategy: 'packages', out: 'api' }]]
};
```

## 3. 컴포넌트 시연 (Storybook 8)

### 3.1 핵심 설계

- **`@storybook/react-vite` framework** 채택. 모노레포가 Vite 기반이므로 webpack 빌더를
  중복 도입하지 않는다(`react-webpack5` 배제 — devDep ~300MB 증가/HMR 저하 회피).
- **CSF3** 패턴. `docs.autodocs: 'tag'` 로 `tags: ['autodocs']` story에 자동 문서 생성.
- **story 수집 glob**(실제 동작 기준):

  ```typescript
  stories: [
    '../../../packages/*/stories/**/*.stories.@(ts|tsx)',
    '../../../packages/*/src/__stories__/**/*.stories.@(ts|tsx)',
  ]
  ```

  `apps/docs/.storybook/` 기준 **3단계 상위**가 모노레포 루트이며, 그 아래 `packages/`에
  접근한다. 패키지별 `stories/`와 컴포넌트 단위 `src/__stories__/`를 함께 수집한다.
  패키지에 stories 디렉토리가 없으면 glob에서 자연히 제외된다.
- **`preview.ts`에 CSS import 없음** — 스타일은 Tailwind 클래스 기반이라 별도 글로벌 CSS가
  불필요(설계결정 §6.1).

### 3.2 story 커버리지 규칙

- **export Component를 가진 패키지(11개)는 각 컴포넌트에 최소 1개 story**.
- **Component가 없는 패키지는 면제**: `grid`(메타 `export {}` 플레이스홀더),
  `grid-license`(`setLicenseKey` 함수 + 타입만). 빈 story 파일은 "no stories" 경고를
  유발하므로 의도적으로 생성하지 않는다.
- **`grid-export`는 Component 0이지만 story 포함** — `exportToExcel`, `exportToCSV`,
  `exportToPdf`, `copyToClipboard`, `printGrid` 5개 함수를 mock Grid + 버튼 UI로 시연하여
  "export 동작 누락"을 시각적으로 감지 가능하게 한다.
- **중복 회피**: 다른 패키지에서 re-export되는 레거시 alias(`TreeGrid`/`ColumnPinGrid` 등)는
  원본 패키지 story에서만 커버하고 별도 story를 만들지 않는다.

### 3.3 대용량 가상화 시나리오 (필수)

`grid-core`에 1000행/5000행 가상화 story를 둔다. `@tanstack/react-virtual` 기반 가상
스크롤을 실제 데이터량으로 시연·검증한다. 활성화 prop:

```typescript
<Grid
  columns={columns}
  data={LARGE_DATA}            // 1000~5000 행
  enableVirtualization={true}
  virtualScrollHeight={600}    // 컨테이너 높이; 미지정 시 400 기본
/>
```

| 스크립트 | 명령 | 기대 |
|----------|------|------|
| `storybook` | `storybook dev --port 6006` | 로컬 시연 서버 |
| `build-storybook` | `storybook build` | `storybook-static/` 생성, exit 0 |

## 4. 시각 회귀 (Playwright)

### 4.1 핵심 설계

- **도구: Playwright** (`@playwright/test` Apache-2.0). Chromatic 미채택. 근거 4가지:
  1. **비용** — Playwright는 OSS(무료 한도 없음). Chromatic 무료 5,000 snapshots/month는
     story × PR 수가 늘면 초과 위험.
  2. **외부 SaaS 비의존** — baseline을 git에 self-host. 서비스 장애·토큰 관리 불필요.
  3. **CI 통합** — 표준 YAML + `npx playwright install` 단일 step. secret 불필요.
  4. **라이선스** — Apache-2.0로 OSS-only 정책에 부합. SaaS 약관 의존 위험 없음.
- **baseline 저장**: Playwright 기본 경로 `tests/visual/storybook.spec.ts-snapshots/*.png`
  (`snapshotPathTemplate` 미설정 → spec 파일명 기준 디렉토리). git commit으로 이력 추적
  (self-hosted, 오프라인 동작 가능).

### 4.2 동작 방식

`tests/visual/storybook.spec.ts`가 **Storybook static 빌드 산출물을 동적으로 읽어**
story id 목록을 얻고, 각 story의 iframe URL
(`/iframe.html?id=<storyId>&viewMode=story`)을 순회하며 스크린샷을 baseline과 비교한다.
Storybook 8 기본 산출은 `index.json`(`.entries`)을 **우선** 읽고, 미존재 시
`stories.json`(v6 legacy `.stories`)로 폴백한다. `type: 'docs'` 엔트리는 제외한다.
story를 추가해도 spec 코드 수정이 불필요하다(동적 수집).

```typescript
// playwright.config.ts (요지)
defineConfig({
  testDir: './tests/visual',
  fullyParallel: false,                 // 스크린샷 재현성 위해 순차 실행
  retries: process.env.CI ? 1 : 0,
  use: { baseURL: 'http://localhost:6006', screenshot: 'only-on-failure' },
  projects: [{ name: 'chromium', use: { ...devices['Desktop Chrome'] } }],
});
```

비교 임계값: `toMatchSnapshot(..., { maxDiffPixelRatio: 0.01 })` — OS/브라우저 렌더링
차이 1% 허용. 가상화/1000행 story(story id에 `virtual`/`1000` 포함)는 렌더 시간이 길어
`networkidle` 대기 timeout을 **60s**로 연장하고, 그 외 story는 5s를 쓴다.

### 4.3 CI 워크플로우

`.github/workflows/visual-regression.yml`:

1. `on: pull_request (branches: [main])`
2. `pnpm install` → `pnpm -F docs build-storybook` → `npx playwright install --with-deps chromium`
   → `pnpm visual:test`
3. 실패 시 `playwright-report/`를 artifact로 업로드.
4. 별도 `block-on-migration-impact` job: PR 라벨이 `migration-impact:high` 또는
   `migration-impact:medium`이고 시각 회귀가 실패하면 `exit 1`로 **PR을 차단**. 라벨이 없는
   PR은 통보만 한다(차단 안 함).

스크립트:

| 스크립트 | 명령 | 용도 |
|----------|------|------|
| `visual:test` | `playwright test` | 회귀 비교 실행 |
| `visual:update-baseline` | `playwright test --update-snapshots` | baseline 갱신(의도적 UI 변경 후) |

baseline 갱신 절차는 `apps/docs/visual-regression.md`에 문서화: 최초 baseline 생성(EC),
CI(ubuntu+chromium) 기준 생성 권장, 차이 발생 시 `maxDiffPixelRatio` 조정.

## 5. 마이그레이션 가이드 (레거시 → `<Grid>`)

`apps/docs/docs/migration/`의 문서 5종. 코드 이전을 *수행*하지 않고 전환 매핑을 안내하는
참조 문서다. 여기에 담긴 매핑 자체가 그리드 API 지식이다.

### 5.1 8개 Grid 변형 → `<Grid>` (`8-variant-table.md`)

| 변형 | `<Grid>` 매핑 | 대상 패키지 | 비고 |
|------|---------------|-------------|------|
| `BaseGrid` | `<Grid mode="client" data columns pagination rowSelection onRowClick .../>` | `grid-core` | |
| `VirtualGrid` | `<Grid enableVirtualization rowHeight={N} containerHeight={N}/>` | `grid-core` | pagination prop 제거 권장 |
| `TreeGrid` | `<TreeGrid ...>` (grid-core legacy alias) | `grid-core` | |
| `ColumnPinGrid` | `<ColumnPinGrid pinLeft pinRight ...>` (legacy alias) | `grid-core` | |
| `EditableGrid` | `<Grid enableEditing keyboardEdit>` + `useChangeTracking` | `grid-renderers` + `grid-pro-tracking` | 조합형 |
| `GroupedHeaderGrid` | `<GroupedHeaderGrid ...>` (re-export) | `grid-pro-header` | 기존 import 그대로 |
| `ChangeTrackingGrid` | compat shim 유지 / `useChangeTracking` 직접 사용 | `grid-pro-tracking` | 기존 코드 변경 불필요 |
| `RangeSelectGrid` | wrapper 유지 / `useCellRange` 직접 사용 | `grid-pro-range` | 기존 코드 변경 불필요 |

**레거시 alias는 `grid-core`에서 정확히 5개만 export**: `BaseGrid`, `VirtualGrid`,
`ColumnPinGrid`, `GroupedHeaderGrid`, `TreeGrid`(각 `*Props` 포함). `EditableGrid` /
`ChangeTrackingGrid` / `RangeSelectGrid`에는 grid-core alias가 **없으며**, 각각
`grid-renderers`+`grid-pro-tracking` 조합, `grid-pro-tracking`, `grid-pro-range`에서
직접 import한다.

### 5.2 DataTable → `<Grid mode="server">` (`dataTable-migration.md`)

서버 페이지네이션 중심의 `DataTable`(별도 아키텍처)을 `<Grid>`로 옮기는 매핑:

| DataTable | `<Grid>` |
|-----------|----------|
| `pageingInfo: { hasNextPage, pageCount, pageNo, pageSize, totalCount, ... }` | `pagination: { pageIndex, pageSize, pageCount }` + `onPageChange` |
| `columnInfos: ColumnInfo[]` | `columns: ColumnDef<TData>[]` (`createColumns(columnInfos)` 헬퍼 또는 직접 정의) |
| `buttonInfo: ButtonInfo` | toolbar slot / 별도 toolbar 컴포넌트 |
| `rowActionInfo: RowActionInfo` | `columns`에 action cell renderer 추가 |
| `permissions` prop | 상위 컴포넌트 조건부 렌더링 / toolbar slot 위임 |
| `listAction(act, value)` 단일 디스패처 | 개별 핸들러로 분리 |

`ColumnInfo`는 `grid-core`에 DataTable 호환 legacy type으로 존재한다. `type: 'custom'`
컬럼의 `customFormatter`는 `ColumnDef.cell` renderer로 매핑한다.

### 5.3 증분 이전 전략 (`incremental-strategy.md`)

- Goal당 사용처 파일 수 상한(≤5)을 지키고, 도메인 유형 + 변형 종류로 그룹핑.
- 권장 순서: **완전 이전 변형 사용처 → 부분 이전 변형 → 미이전 변형**.
- 파일별 독립 이전으로 부분 롤백 가능.

### 5.4 Live demo (`live-demos.md`)

StackBlitz embed ≥3개(basic / virtualized / change-tracking). CSP 차단 환경 대비
각 embed 직후 `<details>` 정적 코드 블록 fallback을 **필수**로 둔다.

### 5.5 Deprecated alias 정책 (`deprecated-aliases.md`)

- grid-core legacy alias 5개를 문서화하고 각 권장 대체 API를 안내.
- alias는 **최소 1 minor version 동안 유지**(즉시 제거 금지) 후 다음 메이저에서 제거.

## 6. 핵심 설계 결정과 근거

### 6.1 글로벌 CSS 파일 미생성 — `customCss: []`, preview.ts CSS import 없음
스타일은 Tailwind 클래스 기반이라 신규 `.css` 파일을 만들지 않는다. Docusaurus
preset-classic은 `theme.customCss`를 **빈 배열**로 두어도 기본 테마로 정상 빌드되고,
Storybook `preview.ts`도 글로벌 CSS를 import하지 않는다. 테마 커스터마이징이 필요하면
Tailwind 단독으로는 Docusaurus 테마 변수를 재정의할 수 없으므로(swizzle/CSS 변수 방식
필요) 별도 작업으로 분리한다.

### 6.2 도구는 OSS-only (상용 SaaS 의존 회피)
문서/시연/회귀 전 도구를 MIT·Apache-2.0로 제한한다. 특히 시각 회귀에서 Chromatic(SaaS)
대신 Playwright(self-host)를 택한 이유는 §4.1의 비용·SaaS 의존·CI 통합·라이선스 4가지.
baseline을 git에 두어 외부 서비스 장애·토큰 관리·snapshot 한도 문제를 원천 차단한다.

### 6.3 API 문서·story의 자동 수집 (수기 유지 최소화)
- TypeDoc은 13개 `src/index.ts`를 entryPoint로 자동 변환 → 패키지가 늘어도 설정만 추가.
- Storybook은 glob 기반 자동 수집 → 새 story 추가 시 main.ts 수정 불필요.
- 시각 회귀는 `stories.json`을 런타임에 읽어 story를 순회 → spec.ts 수정 불필요.

세 지점 모두 "소스가 진실, 문서는 파생"을 강제해 드리프트를 줄인다.

### 6.4 JSDoc 누락을 빌드 실패로 (`failOnWarnings: true`)
공개 API 문서화를 권고가 아닌 게이트로 만든다. export에 JSDoc이 빠지면 `docs:build`가
실패하므로 CI에서 자동 차단된다.

### 6.5 Component 없는 패키지의 story 면제 + 함수 패키지의 예외 포함
빈 story는 도구 경고만 만들고 가치가 없으므로 메타/라이선스 패키지는 면제한다. 반대로
`grid-export`는 Component가 없어도 export 함수 5개의 회귀 가치가 커서 mock UI story로
포함한다 — "면제 가능 ≠ 면제해야 함"을 사용 가치 기준으로 판단한다.

### 6.6 마이그레이션은 안내, 이전 상태를 정확히 표기
가이드 문서는 코드를 옮기지 않고 매핑만 제공하되, 각 변형의 **현재 이전 상태(미이전/부분/
완전)**를 실제 소스 기준으로 표기한다. 이미 re-export·shim·wrapper로 완전 이전된 변형은
"추가 작업 불필요"를 명시해 불필요한 재작업을 막는다.

## 7. 패키지 README + Pro 라이선스

13개 패키지 각각에 `README.md`를 둔다. 구조는 두 갈래:

**MIT 패키지**(`grid-core`, `grid-features`, `grid-export`, `grid-renderers`):
설명 → Installation → Usage(tsx) → Peer Dependencies → License(MIT).

**Pro 패키지**(`grid-pro-*` 7개): 위 구조 + **License Activation 섹션**(필수). 라이선스
활성화는 `grid-license`의 `setLicenseKey`를 사용한다:

```tsx
import { setLicenseKey } from '@topgrid/grid-license';
setLicenseKey('YOUR-LICENSE-KEY');   // 앱 진입점에서 1회. 미활성 시 워터마크 렌더
```

- 각 Pro 패키지 디렉토리에 `EULA.md`를 둔다(플레이스홀더: 라이선스 문의 연락처 안내).
  공유 파일이 아니라 패키지별로 두는 이유는 publish 시 패키지 `files`에 포함되어야 npm에
  노출되기 때문(표준 npm 관행).
- 메타/엔진 패키지(`grid`, `grid-license`)는 용도 설명 + 라이선스 섹션만. `grid`은
  `private`이라 설치 명령 생략하고 "개별 패키지를 직접 사용"을 안내한다.
- **라이선스 API의 정식 이름은 `setLicenseKey`** (`grid-license`의 실제 export).
  `initLicense`는 존재하지 않는다 — 문서 작성 시 주의.

> `createTopgridColumnHelper`, `ColumnInfo` 등은 실제 export 식별자이므로 README/예시에서
> 그대로 사용한다. (옛 `createTomisColumnHelper` 별칭은 clean-break 로 제거됨)
